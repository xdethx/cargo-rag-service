# 2026-06-26 — Milestone 1: Ingestion + Retrieval

## What was built
Offline ingestion (`src/ingest.py`) that loads 5 corpus files (3 Markdown, 1 txt, 1 PDF),
splits them into 23 chunks, embeds them with a local sentence-transformers model, and saves a
FAISS index. Retrieval (`src/rag.py`) that embeds a query and returns the top-k chunks with
source filename and L2 distance score. A CLI harness (`src/search.py`) supporting both a
one-shot argument mode and an interactive REPL. Six passing pytest tests asserting known
questions return the correct source files.

## Key decisions & why

**Offline/online split.** Ingestion runs once when documents change; the API loads the
prebuilt index read-only at startup. This is the standard production pattern — embedding
the whole corpus on every request would be absurdly slow and expensive.

**`sentence-transformers/all-MiniLM-L6-v2`.** Free, runs fully local, no API key. Fast
enough that the pytest fixture (ingest from scratch) completes in ~10 seconds. `get_embeddings()`
lives in `config.py` as the single shared function so ingest and query are guaranteed to use
the same model — if they differ, cosine distances are meaningless.

**FAISS with default `IndexFlatL2`.** Exact nearest-neighbour search, no approximation.
Fine for 23 chunks; even at 10 000 chunks the brute-force L2 scan would be sub-millisecond.
Score reported as L2 distance (lower = more similar), labelled clearly in the CLI output.

**`TextLoader(encoding="utf-8")`.** The corpus contains em-dashes and arrows. Windows
defaults to cp1252/cp1254 and would raise `UnicodeDecodeError` without the explicit
encoding.

**`langchain-community` deprecation warning.** As of langchain-community 0.4.2, the package
prints a DeprecationWarning noting it is being sunset. The concrete migration path (standalone
`langchain-faiss`, standalone document-loader packages) isn't fully available yet; we'll
migrate when a stable replacement lands. Tracked here so it isn't forgotten.

## Deferred for scale
- **FAISS → pgvector or Qdrant** when the corpus grows beyond a few thousand chunks, or when
  metadata filtering (e.g. "only search 2024 documents") is needed.
- **Synchronous ingest script → background job / queue** (Celery, RQ, or a cloud function)
  when documents change frequently and blocking is unacceptable.
- **No caching** — under load, add a query + embedding cache (Redis) to avoid re-embedding
  identical questions.
- **No auth or rate-limit on `/ask`** — add before any public deployment.
- **Naive top-k retrieval** — add reranking (cross-encoder) or hybrid BM25+dense search if
  answer quality is insufficient for edge-case questions.
- **HF Hub warnings** — `HF_TOKEN` env var and enabling Windows Developer Mode (for symlinks)
  would silence the two startup warnings; not worth doing until deployment matters.

# 2026-06-27 — Milestone 2: Generation + FastAPI `/ask`

## What was built
`src/llm.py` — a small factory `get_chat_model()` returns the right LangChain chat client
based on `LLM_PROVIDER`; covers Ollama, Anthropic, any OpenAI-compatible API (Grok/Groq/
OpenRouter), and Gemini. `src/rag.py` gained `build_prompt()` and `generate()`: retrieve
top-k chunks → format them into a grounded system+human prompt → call the LLM → return the
answer **and** the source chunks. `src/main.py` is a FastAPI app with `POST /ask` and `GET
/health` (real LLM liveness probe). 12 tests pass with zero live LLM calls.

## Key decisions & why

**Provider factory in `src/llm.py`, not inline in `rag.py`.** Single-responsibility: the
RAG logic doesn't care which provider is in use, and `get_chat_model()` can be tested or
swapped in isolation. Adding a fifth provider means touching only `llm.py` and `env.example`.

**Grounded prompt by hand, not a LangChain prompt template.** System message: "answer using
ONLY the context; say clearly when it isn't there." Human message: each chunk labelled
`[source: filename]` then the question. Every field is visible and explainable. The "not in
the documents" instruction is critical for a cargo app — a confidently wrong customs answer
causes real problems.

**Index + LLM cached at module level (`get_index`, `get_llm`), warmed by FastAPI lifespan.**
The embedding model takes ~2 s to load; the FAISS index is an in-memory structure. Loading
them per request would dominate latency. Lifespan startup also fails fast if the index is
missing or the provider is misconfigured, before any traffic hits the app.

**LLM injection params on `generate()`** (`index=None, llm=None` default to caches). This
lets `test_generate.py` inject a `CapturingFakeLLM` with zero network calls. The capturing
fake stores `last_messages` so we can assert the context block actually reached the model —
otherwise a passing test would only prove the function ran, not that grounding works.

**502 for LLM errors, 503 for missing index.** These are distinct failure modes with
different remedies (restart Ollama vs re-run ingest). Separate status codes let a calling
service distinguish them without parsing error text.

**`/health` does a real LLM probe.** A health check that says "ok" while Ollama is down is
actively misleading in a compose/k8s readiness-check context. `check_llm()` does a one-shot
`invoke("ping")` — cheap, and the result is labelled `"reachable"` or the exception string.
Cloud callers may want to gate this endpoint or call it sparingly (it uses one token).

## Deferred for scale
- **Streaming** — `llm.stream()` instead of `invoke()` so the user sees tokens as they arrive;
  requires FastAPI `StreamingResponse`.
- **LLM retries / timeouts** — `tenacity` retry wrapper around `llm.invoke()` with exponential
  backoff; set a per-request timeout to avoid hanging on a slow provider.
- **Prompt-injection hardening** — sanitise or size-limit `question` before inserting into
  the prompt; add a system-level injection-resistance instruction.
- **Answer / embedding cache** — cache `(question, k) → result` in Redis under load; cache
  the query embedding to skip re-encoding repeated questions.
- **Auth + rate-limit on `/ask`** — API key check or JWT middleware before any public
  deployment; rate-limit per key to prevent abuse.
- **Streaming `/health`** — the current probe adds latency to every health check; replace
  with a lightweight model-list or metadata call when supported by the provider.

# 2026-06-27 — Milestone 3.1: Deploy prep (Commit A + B)

## What was built

**Commit A — cloud LLM plumbing + container setup.** Groq is reachable today via the
existing `openai_compatible` provider path (`LLM_PROVIDER=openai_compatible`,
`OPENAI_COMPAT_BASE_URL=https://api.groq.com/openai/v1`) — no new provider branch was added.
`GET /health` was made a cheap liveness check (no LLM call, safe for keep-warm pings).
`PORT` is now configurable via env (default 7860, the Hugging Face Docker Space default).
A `Dockerfile` builds a `python:3.11-slim` image with the CPU-only torch wheel, bakes the
embedding model weights at build time, and copies the prebuilt FAISS index from the repo.
The FAISS index was committed to git (removed from `.gitignore`) so it is in the git build
context when HF builds the image remotely. A `.dockerignore` excludes `.env`, `.venv`, tests,
journal, and git history.

**Commit B — endpoint protection.** `POST /ask` now requires `Authorization: Bearer <key>`
matching `RAG_API_KEY` (checked with `secrets.compare_digest`). `slowapi` rate-limits `/ask`
to 10 req/min per real client IP (reads `X-Forwarded-For` from HF's proxy). Four new hermetic
tests cover 401 (no key), 401 (wrong key), 200 (valid key), and 429 (11th request with valid
key from the same IP). 16 tests pass total.

## Key decisions & why

**Groq via `openai_compatible`, not a dedicated `groq` provider.** The service already had a
generic OpenAI-compatible client path (base_url + api_key + model from env). Adding a named
`groq` branch would duplicate that path and violate CLAUDE.md's "never hardcode a provider"
rule. Groq is just configuration, not new code.

**Cheap `/health` (no LLM call).** Keep-warm pings are frequent; a billed `invoke("ping")`
on every ping wastes tokens and can exhaust a free-tier rate limit. An accurate "is the model
responding?" probe is useful but should be opt-in (not the default).

**Shared-secret API key, not JWT.** No end-user identity is needed here — the .NET app calls
the service with a single shared secret. JWT would add key infrastructure for no benefit.
`secrets.compare_digest` avoids timing-based key inference.

**The API key, not the IP rate limit, is the real security boundary.** The rate limit guards
against runaway callers; the key guards against unauthorized callers. An attacker without the
key cannot reach the rate limiter.

**Auth resolves before the rate limiter.** Unauthenticated requests are rejected at the
dependency layer before slowapi's counter increments — they don't consume the client's quota.

**FAISS index committed to the repo; embedding model baked into the image.** HF Docker Spaces
runs containers as a non-root user (uid 1000) with an ephemeral filesystem — runtime downloads
would re-fetch on every cold start and can hit cache-write permission errors. Baking both the
index and the model weights into the image makes the container self-contained and cold starts
fast. `HF_HOME=/app/.cache/huggingface` (under the workdir) + `chmod -R a+rX` ensures the
non-root runtime user can read the weights.

## Deferred for scale
- **Per-user JWT/OAuth** if end users ever call the service directly (not just the .NET app).
- **API key rotation / management** — right now the key lives in a single env var; a key-vault
  or secrets-manager integration would allow rotation without redeployment.
- **Redis-backed shared rate limiter** once there is more than one instance. In-memory slowapi
  counters are per-process and don't enforce a global limit across replicas.
- **Per-user quotas** — the current limit is per-IP, not per-key; under multiple callers with
  different keys, a finer-grained per-key quota would be fairer.
- **Committed index + baked weights is the right trade-off for a small synthetic corpus.** The
  production upgrade path remains a managed vector DB (pgvector / Qdrant) as noted in M1 above.
- **Changing the embedding model requires re-running `python -m src.ingest` AND rebuilding the
  Docker image** (weights are baked in at build time). Flag this explicitly if the model changes.
